Abstract
This study proposes the design and development of a 
forecasting microservice
 
that utilizes Support Vector Regression (SVR) to forecast 
membership
 growth trends using longitudinal feedback data collected from church members over time. While churches globally track attendance and giving, they lack tools to proactively anticipate engagement shifts based on qualitative and behavioral indicators. This research addresses a critical gap: the underutilization of machine learning
 
specifically SVR
 
for modeling sparse, time-series feedback in resource-constrained religious contexts. 
Unlike full Church Management Systems, this system does not collect new member feedback but instead consumes exported CSV/JSON data from existing tools (e.g., Planning Center, Google Sheets, WhatsApp logs). Built as a standalone microservice with a pastor-facing dashboard, it uses Support Vector Regression (SVR) to forecast engagement for cohorts like “Youth Group” or “New Believers,” delivering plain-language alerts (e.g., “Youth engagement may drop 18% in Q3”) without individual profiling. The system is trained on simulated data and evaluated on real anonymized exports from 5 Nairobi churches (100–500 members), with validation via MAE/RMSE against linear regression and ARIMA baselines
 
leveraging its proven robustness on small, noisy datasets and documented design patterns for ethical, lightweight predictive analytics in faith-based organizations.
 Built using React, Node.js, and Python (scikit-learn), the platform delivers actionable, explainable forecasts without surveillance or individual profiling. The study employs Agile methodology across six sprints, with evaluation based on prediction accuracy (MAE, RMSE), usability, and pastoral utility.

DEFINITION OF TERMS
The following technical and domain-specific terms are defined as used in this study.
Membership
 Engagement – The measurable degree of active participation by church members in communal activities, operationalized in this study through weekly attendance headcounts, event RSVP rates, and aggregated monthly satisfaction ratings at the cohort level.
Epsilon-Insensitive Loss Function (ε-SVR) – A mathematical optimization function used in Support Vector Regression that disregards prediction errors falling within a predefined tolerance threshold (ε), thereby preventing the model from overfitting to minor, subjective fluctuations in self-reported feedback data.
Feature Engineering – The systematic process of transforming raw, time-stamped operational records into structured numerical variables that capture predictive temporal patterns, such as rolling averages, linear trend slopes, and participation volatility ratios.
GridSearchCV – An automated hyperparameter optimization technique that exhaustively evaluates all specified combinations of model parameters (e.g., regularization strength, kernel bandwidth) using cross-validation to identify the configuration that yields optimal prediction accuracy on historical data.
Kernel Function – A mathematical transformation that implicitly maps input features into a higher-dimensional space where complex, non-linear relationships between historical behavioral indicators and future engagement outcomes become linearly separable; this study employs the Radial Basis Function (RBF) kernel.
Lagged Feature – A predictive variable derived from historical observations at prior time intervals (e.g., 4-week or 8-week lookback windows), used to encode temporal dependencies and trend momentum for time-series forecasting.
Longitudinal Data – Repeated, time-ordered observations of the same variables collected for consistent cohort groups over an extended period (minimum 24 weeks in this study), enabling the detection of directional trends rather than isolated snapshots.
Mean Absolute Error (MAE) – A model evaluation metric that calculates the average absolute difference between predicted and actual attendance headcounts, providing an interpretable measure of forecast accuracy in terms of actual member counts.
Microservice Architecture – A software design pattern that decomposes the system into loosely coupled, independently deployable services (e.g., a Python-based SVR forecasting engine separate from the Node.js web API), enabling isolated scaling, easier maintenance, and technology-specific optimization.
Personally Identifiable Information (PII) – Any data that can uniquely identify an individual member (e.g., names, phone numbers, national IDs, or individual attendance logs); structurally excluded from all analytical tables in this system to enforce group-level anonymization and comply with Kenya’s Data Protection Act (2019).
Radial Basis Function (RBF) Kernel – The specific kernel function selected for the SVR model that measures similarity between data points based on Euclidean distance, proving highly effective for capturing non-linear, context-dependent engagement dynamics in small, noisy datasets.
Role-Based Access Control (RBAC) – A security mechanism that restricts system functionality based on authenticated user roles (Admin, Pastor, Usher), ensuring that data submission, forecast viewing, and system configuration permissions are strictly segregated according to operational responsibility.
Root Mean Squared Error (RMSE) – A secondary model evaluation metric that squares prediction errors before averaging and taking the square root, placing greater penalty on large forecast deviations (e.g., sudden attendance drops) to highlight critical accuracy failures.
Support Vector Regression (SVR) – A supervised machine learning algorithm adapted for continuous outcome prediction that identifies an optimal regression hyperplane maximizing the margin of tolerance around actual data points, specifically chosen for its robust generalization on small, irregularly sampled longitudinal datasets.
System Usability Scale (SUS) – A standardized 10-item questionnaire administered to end-users to quantitatively assess the dashboard’s learnability, efficiency, and satisfaction, with a score ≥68 indicating above-average usability for non-technical pastoral staff.
TimeSeriesSplit – A cross-validation strategy that partitions chronological data into sequentially expanding training windows and fixed future test sets, preventing temporal data leakage and ensuring the SVR model is evaluated under realistic forward-predicting conditions.
Walk-Forward Validation – The overarching time-series evaluation methodology employed in this study, wherein the model is trained on historical data and tested exclusively on subsequent unseen periods, mirroring real-world deployment and preventing optimistic accuracy inflation

1.2 Background
 
The integration of data-driven decision-making into organizational operations has become a hallmark of digital transformation across banking, healthcare, education, and e-commerce sectors. In these domains, predictive analytics enables proactive planning, resource optimization, and personalized service delivery (Heeks, 2023). For instance, retail platforms use time-series forecasting to anticipate demand, while healthcare systems predict patient no-shows using historical appointment data (Zhang et al., 2023; Shatte et al., 2021). Despite these advances, religious 
organizations, 
particularly in developing regions
 
remain largely reactive in their administrative and pastoral approaches, relying on intuition, anecdotal observation, or lagging indicators such as attendance drops or formal exit notices.
Religious organizations globally collect operational data: weekly attendance, event participation, and increasingly, digital feedback. However, this data is rarely structured or leveraged for forward-looking insight. Instead, it serves retrospective reporting
 
answering “what happened?” rather than “what might happen next?” This limitation is especially consequential in contexts where pastoral capacity is stretched thin, and early intervention could prevent disengagement before it becomes irreversible. Chaves and Anderson (2022) found that 70% of church dropouts leave silently, often due to unmet relational or spiritual needs that went undetected until it was too late. Similarly, the Kenya National Council of Churches (KNCC, 2024) reports that 68% of 
new believers in Kenya disengage within six months, citing lack of follow-up as the primary reason.
Machine learning offers a promising pathway to bridge this gap. Among algorithms suitable for small, noisy, and irregularly sampled datasets
 
common characteristics of church feedback
 
Support Vector Regression (SVR) stands out for its theoretical robustness and practical performance. Unlike deep learning models that require thousands of data points, SVR operates effectively with limited observations by mapping input features into high-dimensional spaces using kernel functions, then identifying optimal hyperplanes that minimize prediction error while maximizing generalization (Smola & Schölkopf, 2004).
Despite this suitability, SVR remains virtually unexplored in religious or 
membership
 contexts. Existing Church Management Systems (CMS) like Planning Center and Breeze offer rich dashboards for historical metrics but provide no predictive capabilities (Mullins, 2021). Academic research on faith-based organizations focuses on theological discourse or sociological trends
 
not applied predictive modeling (Chen et al., 2023).
This disconnect is exacerbated in Global South contexts like Kenya, where digital adoption in churches is growing but remains fragmented. Many 
populations
 use WhatsApp groups for announcements, Google Forms for event RSVPs, and paper registers for attendance
 
creating siloed, unstructured data that cannot be modeled. Even when digital tools are adopted, they lack integration between feedback, attendance, and behavioral patterns.
The opportunity, therefore, lies in designing a lightweight, interpretable, and ethically grounded predictive microservice that leverages SVR to forecast 
membership
 engagement from existing historical data exports. Such a system would not replace human shepherding but extend it
 
enabling pastors to identify at-risk groups weeks in advance and deploy timely, grace-filled interventions.
The opportunity, therefore, lies in designing a lightweight, interpretable, and ethically grounded predictive system that leverages SVR to forecast

 Statement of the problem
Religious organizations in Nairobi lack predictive tools to anticipate population growth trends, relying instead on lagging indicators like attendance drops or exit interviews
 
often too late for meaningful intervention. While digital record-keeping has improved data collection, existing Church Management Systems provide only historical reporting, not forward-looking insights. This reactive approach results in preventable attrition, wasted ministry resources, and diminished community cohesion.
Current analytics in religious platforms are limited to descriptive dashboards (e.g., “Total Attendance: 240”) with no integration of qualitative feedback or temporal modeling. Academic literature on 
membership
 behavior focuses on sociology or theology, not applied machine learning (Chaves, 2020; Mullins, 2021). Meanwhile, commercial predictive analytics tools are cost-prohibitive and designed for retail or marketing
 
not pastoral care.
Support Vector Regression (SVR) offers a theoretically sound alternative: it performs well on small, non-linear datasets and provides robust generalization
 
ideal for churches collecting modest amounts of feedback over time. Yet no existing system applies SVR to longitudinal pastoral data for growth forecasting.
This study therefore investigates how Support Vector Regression can be leveraged to build an affordable, interpretable, and ethically grounded forecasting microservice that helps churches anticipate population growth trends from existing historical data exports.
1.

 Proposed Solution
 
This research proposes a standalone SVR microservice integrated with a React-based pastoral dashboard that ingests existing church attendance and membership records, applies feature engineering to derive temporal predictive variables, and generates group-level population growth forecasts for church leadership. The system comprises four integrated components: a secure data upload module that accepts CSV and Excel exports from churches' existing digital tools; a 
feature engineering pipeline that transforms raw inputs into time-lagged variables such as rolling averages, trend slopes, and participation volatility measures; an SVR forecasting engine implemented via Python's scikit-learn library and exposed through a FastAPI REST endpoint; and a pastoral dashboard built with React.js that visualizes predicted trends and generates actionable, plain-language alerts for church leaders
.

1.
8
.1 
Tools and Techniques
This study will utilize the following technologies, each selected for its suitability to the research goals, compatibility with lightweight deployment, and alignment with open-source, cost-effective development:
Frontend: React.js (with Vite and Tailwind CSS)
The frontend will be a pastor- and administrator-facing dashboard built with React.js, Vite, and Tailwind CSS. Its sole purpose is to allow authorized users to upload their church’s existing historical data exports (e.g., CSV files from Planning Center or Google Sheets), view group-level forecasts, and receive plain-language, actionable alerts. The interface will be mobile-responsive and designed for non-technical users, ensuring accessibility on any modern smartphone or computer browser.
Backend: Node.js with Express.js
A lean Node.js/Express.js backend will serve as the system’s orchestration layer. It will manage user authentication, handle the secure upload of historical data files, trigger the feature engineering pipeline, and coordinate requests between the frontend and the 
dedicated Python ML microservice. This API layer ensures a clean separation of concerns and a responsive user experience.
 Its non-blocking I/O model ensures efficient performance even on low-cost hosting platforms.
Machine Learning Engine: 
Python with scikit-learn (as a Microservice)
The core predictive capability is a standalone Python microservice built with FastAPI and scikit-learn. This service is entirely decoupled from the main web application. It receives pre-processed, group-level feature vectors from the Node.js orchestrator, loads the trained SVR model, generates forecasts for 4, 8, and 12-week horizons, and returns predictions with confidence intervals. This microservice architecture allows for independent development, testing, and scaling of the ML component without impacting the rest of the system.
 
The SVR model will be trained on engineered features such as rolling averages, trend slopes, and participation volatility to forecast group-level engagement across three selectable horizons: 1 week (immediate operational planning), 4 weeks (monthly pastoral planning), and 8–12 weeks (quarterly ministry strategy), with shorter horizons yielding higher prediction accuracy.
Database: PostgreSQL
PostgreSQL will store all system data, including user accounts, church group definitions, engineered features, forecast results, and alert logs. The schema is explicitly designed to enforce the project’s ethical mandate: no personally identifiable information (PII) is stored in any analytical table. All records in the features, forecasts, and alerts tables are linked exclusively to anonymized group_ids (e.g., “Youth Group”), making individual profiling structurally impossible
.
Deployment: Render (free tier)
The full stack (frontend, backend, and database) will be deployed on Render, a cloud platform offering free-tier hosting for web applications. Render supports seamless integration of React, Node.js, and PostgreSQL, automatic HTTPS, and Git-based continuous deployment. This eliminates infrastructure costs and simplifies maintenance
 
key considerations for resource-constrained religious organizations.
Evaluation Metrics: MAE, RMSE, and System Usability Scale (SUS)
The system’s effectiveness will be evaluated using both technical and human-centered metrics. Mean Absolute Error (MAE) and Root Mean Squared Error (RMSE) will quantify the SVR model’s forecasting accuracy against baseline models (linear regression, ARIMA). The System Usability Scale (SUS), a standardized 10-item questionnaire, will assess the dashboard’s intuitiveness and actionability for pastors and administrators.

Support Vector Regression (SVR) was selected over alternatives like deep learning or ARIMA because of its proven robustness on the small, sparse, and noisy datasets that characterize church

1.
9
 Scope
 
This study focuses on evangelical faith organizations in Nairobi County, Kenya, and centers on the design, development, and evaluation of an SVR-based predictive analytics system for forecasting population growth trends from existing longitudinal membership and attendance records. The target organizations are evangelical Christian churches with 100 to 500 active members that already maintain digital attendance and membership records in spreadsheets, Google Sheets, or a Church Management System from which data can be exported in CSV or Excel format.
 
The study targets a purposive sample of five to ten such churches in Nairobi County. Selected churches must have maintained consistent digital records for a minimum of 24 weeks, have defined membership cohort groups (e.g., youth, new believers, young families), and have a designated church administrator or pastor who can participate in requirements elicitation interviews and usability evaluation. The demographic profile of target churches is evangelical Protestant, urban Nairobi setting, congregation size 100 to 500 members, and existing use of at least one digital record-keeping tool.
The system will support five core functions: secure ingestion of historical attendance and membership data via CSV/Excel upload; feature engineering pipeline computing lagged temporal variables; SVR-based forecasting of group-level population growth or decline; proactive alerting for decline risk (greater than 15% drop) and growth acceleration; and visualization of trends and forecasts through a pastoral dashboard.
 
The prototype will be developed as a web-based application using React, Node.js, Python (scikit-learn), and PostgreSQL, deployable on free-tier cloud platforms. The scope explicitly excludes individual member profiling, personal-level predictions, financial or giving forecasting, native mobile app development for iOS or Android, 
real-time attendance capture (e.g., biometric check-in), and integration with external live Church Management System APIs. The system relies on periodic batch uploads of historical data rather than live data feeds. Member-facing data entry interfaces are also excluded from the current scope; any future implementation of live member input will be addressed as a recommendation in Chapter 4. The resulting system serves as a proof-of-concept demonstrating the feasibility, accuracy, and pastoral utility of SVR-driven population growth forecasting in resource-constrained faith organizations. The prototype will not go live in production.

2.1 Introduction
This chapter critically reviews existing literature on predictive analytics in organizational contexts, with a focused examination of machine learning applications in social forecasting, 
membership
 dynamics, and time-series modeling using Support Vector Regression (SVR). The review is structured across six thematic areas: the theoretical foundations of predictive modeling in social systems; empirical studies on

2.2 Theoretical Review
2.2.1 Time-Series Forecasting in Human Behavior
Predictive modeling of human behavior relies on the foundational assumption that past patterns contain reliable signals about future states. In time-series analysis, this principle is formalized through a range of approaches: autoregressive models that weight recent observations most heavily, moving average models that smooth noise by averaging across windows, and machine learning regressors that capture non-linear relationships between historical features and future outcomes (Hyndman & Athanasopoulos, 2021). Unlike physical systems whose behavior is governed by deterministic equations, human behavioral data is characteristically sparse, irregularly sampled, and noisy properties that challenge classical statistical models which assume regular intervals and normally distributed residuals.
Machine learning approaches, particularly kernel-based methods such as Support Vector Regression (SVR), address these limitations directly by mapping input features into high-dimensional spaces where non-linear relationships become linearly separable (Smola & Schölkopf, 2004). This theoretical mechanism makes SVR especially well-suited to contexts like 
church member feedback, where data points are infrequent for example, monthly satisfaction surveys and inherently subjective. A member who rates their engagement as 3 out of 5 in one week and 4 out of 5 the following week may not have experienced a meaningful change in actual participation; the SVR model's epsilon-insensitive loss function is specifically designed to treat such minor fluctuations as noise rather than signal, producing more stable and interpretable forecasts than models that penalize every deviation.
The application of time-series forecasting to human organizational behavior has expanded significantly in the past decade, moving from financial and industrial domains into social systems including healthcare adherence prediction (Shatte et al., 2021), educational performance forecasting (Al-Rahmi et al., 2024), and employee retention modeling. Each of these applications shares a common theoretical backbone: the identification of leading behavioral indicators actions or attitudes that precede a significant outcome by several weeks or months which can be detected early enough to enable intervention. This study applies the same theoretical framework to 
membership
 engagement, treating weekly attendance rates, event participation ratios, and monthly satisfaction scores as leading indicators of group-level disengagement trajectories.
2.2.2 The Role of Longitudinal Data in Early Warning Systems
Longitudinal data  repeated observations of the same variables for the same subjects at consistent intervals over time is the essential raw material for early warning systems in social organizations. The diagnostic power of longitudinal data lies in its capacity to reveal trajectories rather than snapshots: a single low attendance figure is ambiguous, but a declining trend across six consecutive weeks is a statistically meaningful signal that warrants pastoral attention. In healthcare, weekly mood and symptom logs have been shown to predict patient relapse up to three weeks in advance, enabling clinicians to schedule preventive interventions before a crisis occurs (Shatte et al., 2021). In education, cumulative quiz performance records forecast final course outcomes with sufficient lead time for targeted academic support (Al-Rahmi et al., 2024).
These systems share a fundamental design principle that this study adopts: lagged features derived from longitudinal records including rolling averages, linear trend slopes computed across recent windows, and volatility measures serve as the primary predictors for future behavioral outcomes. The rolling average captures stable central tendency; the trend slope captures directionality 
(whether the trajectory is improving or declining); and the volatility measure captures consistency (whether a group's engagement is stable or erratic). Churches already collect the raw inputs for these features informally through attendance registers, event RSVPs, and post-service feedback, but rarely structure them into model-ready longitudinal datasets. The theoretical contribution of this study lies in demonstrating that this existing, underutilized data is sufficient when properly engineered to power a meaningful predictive system without requiring additional data collection infrastructure.

2.3.1 SVR in Healthcare Adherence Monitoring: Shatte et al. (2021)
The most structurally analogous SVR application to the proposed study is documented by Shatte et al. Shatte et al. The study explicitly recommends SVR for behavioral forecasting contexts where datasets contain fewer than 200 training observations the precise regime applicable to small-to-medium

Al-Rahmi et al. The study is particularly relevant to the current work because its feature engineering strategy directly parallels the approach proposed here: rolling averages of recent quiz scores and linear trend slopes computed across 8-week windows were the primary 
predictors, with the SVR model's RBF kernel capturing the non-linear relationship between mid-semester engagement patterns and final outcomes. Al-Rahmi et al. (2024) demonstrated that this multi-feature approach with temporal lagging outperforms univariate approaches that rely solely on the most recent observation, confirming that the additional engineering complexity is methodologically justified. This finding supports the decision in the current study to derive four engineered features
:
 rolling average, trend slope, RSVP rate, and feedback average rather than using raw attendance counts directly as model inputs.

The closest documented precedent for predictive analytics in a

2.4 Integration and Architecture
2.4.1 Microservice Architecture for Machine Learning Integration
The standard architectural pattern for integrating machine learning models into web applications is the microservice approach, in which the ML model is deployed as an independent service that communicates with the main application layer via a REST API (Newman, 2023). This pattern is especially appropriate for SVR-based systems because it allows the Python-based scikit-learn model to run in its own runtime environment, fully isolated from the Node.js web application layer. This isolation prevents library version conflicts between Python ML dependencies and Node.js 
web dependencies, enables independent redeployment of the model without interrupting the web service, and allows the ML service to be scaled separately if prediction demand increases. Zhang et al. (2023) employed this exact pattern in their retail SVR demand forecasting system, deploying the SVR engine as a Flask microservice called by a Node.js orchestration layer, and reported that the separation of concerns reduced debugging time by approximately 40% compared to an earlier monolithic implementation. The current study adopts the same microservice architecture, using FastAPI in place of Flask for its superior asynchronous performance and automatic OpenAPI documentation generation.
A monolithic alternative embedding Python within the Node.js application via child process spawning  was considered and rejected for three reasons. First, it couples the machine learning and web application release cycles, meaning a model update requires redeployment of the entire application. Second, it degrades response time when model retraining is triggered during active user sessions, as both processes compete for the same compute resources. Third, it significantly complicates deployment on free-tier cloud platforms such as Render, which allocate isolated compute containers per service. The microservice pattern eliminates all three concerns at the cost of minimal inter-service network latency, typically 10 to 50 milliseconds for local calls.
2.4.2 Three-Tier Client-Server Architecture
The three-tier client-server architecture separating the presentation layer, application layer, and data layer into distinct tiers is the established pattern for web-based information systems and provides the structural framework for this project (Heeks, 2023). The presentation tier, built with React.js, handles all user interaction including attendance form submission, feedback surveys, and dashboard visualization. The application tier, built with Node.js and Express.js, manages authentication, business logic, data validation, and orchestration of calls to the Python ML microservice. The data tier, implemented with PostgreSQL, provides persistent, ACID-compliant relational storage for the longitudinal time-series records required for model training and the forecast and alert records required for dashboard display.
This architecture is extensible without structural change: additional congregation groups, new input feature types, or alternative ML models can be added at the application layer by creating new API routes and retraining without modifying the frontend or database schema. The separation of 
concerns also enables targeted testing the React frontend can be tested independently of the API, and the Python ML service can be unit-tested independently of both which directly supports the structured testing strategy described in Chapter 4.
2.4.3 Feature Engineering Pipeline Design
The integration of raw operational records with a machine learning forecasting engine requires an intermediate data transformation stage. In comparable implementations reviewed by Khan et al. (2022) and Al-Rahmi et al. (2024), this stage operates as a nightly batch process: raw records accumulated during the operational day are retrieved from the database, transformed into lagged features through rolling computations, and written to a dedicated features table from which the model reads at both training and inference time. This design decouples data ingestion from model operation, ensuring that pastor-facing forecast requests are served in milliseconds from pre-computed features rather than requiring computationally expensive real-time feature calculation on every request. The current study adopts this batch pipeline architecture, implemented as a scheduled Python script triggered by a Node.js cron job every Sunday at 02:00 East Africa Time, processing the preceding week's records for all registered congregation groups.
2.5 Support Vector Regression in Time-Series Modeling
2.5.1 Theoretical Advantages for Sparse Datasets
Support Vector Regression (SVR) is a kernel-based supervised machine learning algorithm that extends the Support Vector Machine framework to continuous regression tasks. The foundational contribution of SVR is the epsilon-insensitive loss function, introduced by Drucker et al. (1997), which ignores prediction errors smaller than a user-defined threshold ε and applies a linear penalty only to errors that exceed this threshold. This property is of direct practical significance in the current study: minor week-to-week fluctuations in attendance headcount or satisfaction ratings attributable to factors such as public holidays, weather, or individual respondent mood fall within the epsilon tube and contribute zero loss, preventing the model from overfitting to noise. Only sustained directional changes in engagement, which represent genuine behavioral signals, drive the model's learning.
A second critical advantage is SVR's behavior under conditions of limited training data. Unlike deep neural networks, which require thousands of labeled observations to generalize reliably, SVR 
achieves strong generalization from small datasets by constructing a maximum-margin hyperplane in a high-dimensional kernel-induced feature space (Smola & Schölkopf, 2004). The Radial Basis Function (RBF) kernel, which measures similarity between observations as a Gaussian function of their Euclidean distance in the original feature space, is particularly effective for capturing the non-linear, context-dependent relationships between lagged behavioral indicators and future engagement outcomes. This theoretical suitability for small, noisy, non-linear datasets
 
all characteristics of 
membership
 feedback data makes SVR a more appropriate choice than linear regression, which assumes linearity and normality, or ARIMA, which relies on a single univariate time series rather than the multi-feature inputs available in this system.
2.5.2 Practical Implementations in Social Forecasting
Recent empirical applications confirm SVR's practical suitability for social forecasting tasks with the data profile relevant to this study. Zhang et al. (2023) used SVR to forecast retail demand from sparse weekly sales logs, outperforming ARIMA by 22% in Mean Absolute Error on a dataset of 78 weekly observations per product category demonstrating that SVR achieves superior accuracy over classical time-series methods even when training data is limited. Khan et al. (2022) applied SVR to predict energy load from irregular smart-meter readings, achieving 94% accuracy with only eight weeks of historical data, directly validating SVR's viability for the short-history scenario faced by 
populations
 that have only recently adopted digital record-keeping. In the social domain most analogous to church feedback, Al-Rahmi et al. (2024) demonstrated a 31% MAE reduction for SVR over linear regression when the training set contained fewer than 200 observations
 
a finding that establishes a concrete performance benchmark against which this study's results can be evaluated.
Taken together, these applications confirm a consistent pattern: SVR thrives precisely in the conditions that characterize 
membership
 feedback data datasets that are small, self-reported, multi-featured, and temporally ordered. Despite this strong evidential basis for SVR's suitability, no published study has applied SVR to 
membership
 engagement data for the purpose of generating forward-looking pastoral alerts. This absence constitutes the primary research gap that the current study addresses.
2.6 Critical Analysis of Existing Literature
A critical synthesis of the literature reviewed in this chapter reveals three persistent structural limitations that explain why predictive analytics has not yet been successfully applied to 
membership
 engagement in resource-constrained African contexts, and that collectively define the research opportunity addressed by this study.
The first limitation is a Western-centric design bias. The most methodologically sophisticated work on 
membership
 analytics
 
including the LifeWay Research (2024) pilot and the predictive retention models reviewed by Chaves and Anderson (2022) assumes operational contexts characterized by stable internet connectivity, high digital literacy among church staff, existing CRM infrastructure, and annual technology budgets exceeding USD 10,000. These conditions describe approximately 15% of global 
populations
, predominantly in North America and Western Europe (ITU, 2024). They do not describe the 82% of Kenyan 
populations
 that, according to Mugambi and Theuri (2021), currently rely on paper registers or WhatsApp groups as their primary record-keeping infrastructure. Analytical tools designed for the former context are structurally inapplicable to the latter without fundamental architectural redesign.
The second limitation is tool fragmentation. Even in the minority of African churches that have adopted digital tools, these tools operate as disconnected silos: attendance may be tracked in a Google Sheet, event RSVPs collected through a WhatsApp poll, and member feedback gathered through an informal post-service conversation that is never recorded (Nwosu & Mwangi, 2025). Without a unified data architecture that aggregates these streams into a structured longitudinal dataset, no forecasting algorithm however theoretically sophisticated can generate meaningful predictions. The integrated data collection layer proposed in this study directly addresses this fragmentation by providing a single platform for attendance logging, RSVP tracking, and structured feedback collection, with all inputs automatically aggregated at the group level for feature engineering.
The third limitation is methodological overkill in proposed solutions. Where academic researchers have proposed ML-based approaches for faith-based organizations, they have predominantly suggested deep learning architectures recurrent neural networks, transformer-based sequence models that require thousands of training observations, GPU compute resources, and specialist 
technical expertise to deploy and maintain (Chen et al., 2023). As Nwosu and Mwangi (2025) observe, complexity is the principal barrier to technology adoption in resource-constrained settings. SVR, by contrast, requires no GPU, operates effectively with 50 to 200 training observations, is fully implemented in the widely-documented scikit-learn library, and produces interpretable outputs that non-technical church administrators can act upon without statistical training.
2.7 Research Gaps
The literature reviewed in this chapter establishes three specific, evidence-grounded research gaps that the proposed study is uniquely positioned to address.
Contextual Applicability Gap. Existing predictive models for 
membership
 engagement have been developed for and validated exclusively in well-resourced Western churches with stable digital infrastructure, high staff digital literacy, and existing CRM systems. No validated predictive model exists for small-to-medium evangelical 
populations
 in urban Sub-Saharan Africa, where data is sparse, tools are fragmented, and budgets are constrained (Mugambi & Theuri, 2021; KNCC, 2024). This study addresses the gap by designing a system calibrated specifically for the Nairobi 
membership
 context, with data requirements, infrastructure assumptions, and cost constraints appropriate to that setting.
Integrated Data Architecture Gap. No existing system unifies attendance records, event participation logs, and member feedback ratings into a single structured longitudinal dataset suitable for machine learning in a faith-based context. Churches collect the raw inputs for such a dataset but lack the integration architecture to make them model-ready (Nwosu & Mwangi, 2025). This study addresses the gap by developing an integrated data collection and feature engineering pipeline that aggregates all three input streams at the group level and automatically computes the lagged features required for SVR training.
Lightweight Methodology Gap. Support Vector Regression, despite its proven efficacy on sparse, noisy, self-reported datasets analogous to church feedback (Shatte et al., 2021; Al-Rahmi et al., 2024; Zhang et al., 2023), has not been applied to 
membership
 engagement forecasting. Current FaithTech solutions are either too simple descriptive dashboards that cannot anticipate future trends
 
or too complex deep learning systems requiring data volumes and compute resources 
unavailable to small 
populations
. This study fills the gap by implementing SVR as an interpretable, lightweight alternative that operates effectively within the resource constraints of its target context.

This chapter has reviewed literature across four thematic
 
areas theoretical foundations of predictive modeling in social systems, empirical studies on 
membership
 engagement and retention, machine learning applications for behavioral forecasting in non-commercial domains, and SVR as a methodological framework for sparse longitudinal data to establish the evidence base for the proposed study. The synthesis reveals a coherent and specific opportunity that existing research has not addressed.
Theoretically, the literature establishes that SVR's epsilon-insensitive loss function and kernel-based feature mapping make it uniquely suited to the sparse, noisy, and non-linear nature of self-reported behavioral data (Smola & Schölkopf, 2004; Drucker et al., 1997). Empirically, healthcare and education applications demonstrate that SVR achieves meaningful predictive accuracy  MAE reductions of 22–31% over linear baselines with as few as 8–12 weeks of training data (Zhang et al., 2023; Al-Rahmi et al., 2024; Shatte et al., 2021). These findings directly validate the technical feasibility of applying SVR to 
membership

The critical gap this research fills sits at the intersection of three absences identified in the review. First, no study has applied SVR or any supervised regression model  to 
membership

The closest analogues operate in secular domains (education, retail, healthcare) with different data types, stakeholder needs, and ethical constraints. Third, no published solution is designed for the affordability, interpretability, and offline-capability constraints of small-to-medium African 
populations
. Commercial CMS platforms serve well-resourced Western churches; academic proposals assume cloud infrastructure and large datasets; neither serves the 82% of Kenyan churches that currently rely on paper registers or WhatsApp.
By developing an SVR-based predictive system that is purpose-built for this specific intersection group-level forecasting, voluntary feedback, open-source tooling, and Nairobi-context deployment this study makes a focused and original contribution to both ICT4D and FaithTech. 3.3.3 Operational Feasibility
The system addresses a well-documented and urgent operational need in religious organizations. The proposed system converts this reactive cycle into a proactive one by providing 4 to 12-week advance warning of group-level engagement decline.
Operationally, the system is designed for adoption by non-technical church staff. The mobile-responsive web interface requires no app store installation and functions on any modern smartphone browser. The attendance logging form is designed for completion by ushers during or immediately after a service a two-minute task requiring no technical training. 3.3.4 Legal and Regulatory Feasibility
The project adheres to the legal and ethical standards applicable to personal data processing in Kenya. All data collection and processing activities are designed to comply with the principles established by Kenya's Data Protection Act (2019), specifically: purpose limitation data is collected solely for the purpose of generating group-level pastoral engagement forecasts and is not used for financial analysis, marketing, or any other secondary purpose; data minimization only the minimum data required for group-level modeling is retained (attendance headcounts, aggregated satisfaction ratings, and RSVP counts), with no names, national identification numbers, phone numbers, or other personal identifiers stored in any analytical table; and integrity and confidentiality  all data transmissions are secured via HTTPS and database access is restricted by role-based controls enforced at the API layer.
The system's most critical legal safeguard is the structural exclusion of personally identifiable information from the machine learning pipeline. The features, forecasts, and alerts tables contain only group-level identifiers; no individual member_id exists in any analytical table. This architectural design means that even if the database were compromised, no individual member could be identified or profiled from the analytical data. The design also ensures compliance with the KNCC (2024) pastoral confidentiality guidelines applicable to member data in Kenyan evangelical churches.

All seven participants met these criteria and consented to participation prior to data collection.
Instrument 2: Structured Questionnaire (Convenience Sampling)
For the structured questionnaire phase, convenience sampling was employed to reach a broader group of church staff and leaders. The questionnaire was distributed digitally via Google Forms. A total of seven complete responses were received, representing a 100% completion rate among participants who accessed the form. While the sample size is modest, it is appropriate for the descriptive and confirmatory purpose of this instrument to quantify the frequency of digital tool usage, satisfaction levels, and feature preferences already identified qualitatively through interviews. This sample size is consistent with comparable requirements-elicitation studies in small-scale FaithTech and ICT4D research (Mugambi & Theuri, 2021).

This study was conducted in accordance with the ethical standards applicable to research involving human participants at Jomo Kenyatta University of Agriculture and Technology. The following ethical safeguards were observed throughout the data collection, system design, and evaluation phases of the project.
Informed Consent. All interview and questionnaire participants were provided with a written participant information sheet prior to data collection, explaining the purpose of the study, the voluntary nature of participation, and the right to withdraw at any time without consequence. Written consent was obtained from all interview participants before each session. Questionnaire 
respondents were presented with a consent statement on the first page of the Google Form, and submission was treated as implicit consent.
Anonymity and Confidentiality. Interview participants are identified in this report only by role (e.g., Pastor 1, Administrator 2) with no names, church names, or denominational identifiers disclosed. Questionnaire responses were collected anonymously
 
no email addresses or IP addresses were recorded. Raw interview transcripts and questionnaire response files are stored on a password-protected device accessible only to the researcher and are not included in public-facing appendices.
Group-Level Prediction and Non-Profiling Guarantee. A critical ethical commitment of this research is that the SVR forecasting model operates exclusively on group-level aggregated data. No personally identifiable information (PII)
 
including names, phone numbers, national ID numbers, or individual attendance records is stored in or passed to the feature engineering pipeline or the SVR model. All data entered into the system is aggregated at the cohort level (e.g., 'Youth Group,' 'New Believers') before feature computation. This design choice was validated during requirements elicitation, where 100% of participants stated zero tolerance for any system that profiles individuals. The database schema enforces this guarantee structurally: the features, forecasts, and alerts tables contain only group_id foreign keys; no member_id column exists in any analytical table.
Voluntary Feedback and Data Minimization. Member-submitted feedback (sentiment ratings and optional text reflections) is entirely voluntary. No member is required to submit feedback, and non-submission does not affect their participation in group activities. Optional text feedback fields are stripped of any identifying language at the API layer before storage, using server-side validation that rejects entries containing phone number patterns or proper nouns. This design ensures the system collects only the minimum data necessary to generate group-level forecasts, consistent with the data minimization principle of Kenya's Data Protection Act (2019).
Compliance with Kenya's Data Protection Act (2019). The system design adheres to the principles of Kenya's Data Protection Act (2019), specifically: purpose limitation (data is collected solely for pastoral forecasting, not marketing or financial analysis); data minimization (only attendance counts, aggregated ratings, and RSVP counts are stored); storage limitation (historical data older than 36 months may be archived or deleted without affecting model accuracy); and integrity and 
confidentiality (HTTPS encryption for all data transmission; PostgreSQL role-based access preventing unauthorized queries). While this prototype does not process data at a scale requiring formal registration with the Office of the Data Protection Commissioner, the design patterns adopted ensure the system is compliant from inception.
3.5 Data Analysis
3.5.1 Key Findings from Interviews
Thematic analysis of the seven transcribed interviews identified four dominant themes that directly shaped the system's functional requirements and design constraints.
The first theme was fragmented data ecosystems. All seven participants described managing attendance, event, and feedback data across disconnected platforms with no integration between them. Pastor 2 stated that attendance was tracked in one spreadsheet, event RSVPs in a separate Google Form, and member feedback through informal WhatsApp conversations, making it impossible to see patterns across the three data streams. This finding directly motivated the integrated data collection architecture described in Section 3.8, which unifies all three inputs within a single platform.
The second theme was the reactive nature of current pastoral care. All seven participants reported becoming aware of member disengagement only after the individual had already departed silently. Administrator 1 noted spending approximately three hours each Monday compiling attendance records from multiple sources, leaving no time for trend analysis. This reactive cycle was universally identified as the primary operational gap the proposed system should address, confirming the 4 to 12-week forecast horizon as the most actionable prediction window.
The third theme was strong demand for predictive capability with a specific forecast horizon preference. All seven participants expressed enthusiasm for a system that could identify at-risk groups before disengagement became departure. Pastor 3 explained that a 1 to 3-month forecast horizon would allow sufficient time to organize targeted pastoral interventions such as reconnection dinners or personal follow-up calls, while weekly predictions were considered too volatile and annual predictions too distant for actionable planning.
The fourth theme was non-negotiable ethical boundaries around individual privacy. All seven participants stated without exception that any predictive system must operate at the group level only and must never identify or flag individual members. This requirement is architecturally enforced in the system design through the structural exclusion of individual member identifiers from all analytical database tables.

3.7 Requirements Analysis and Modelling
The requirements gathered in Section 3.6 were analyzed to identify inter-module dependencies, potential conflicts, and architectural constraints. The analysis confirmed a strict dependency ordering: data must be ingested through Module 1 before features can be computed by Module 3; Module 4 can only generate forecasts after Module 3 has computed sufficient lagged features; and Module 5 can only display forecasts after Module 4 has produced them. This ordering is enforced operationally through the weekly cron job scheduling described in FR3.4 and the sequential pipeline execution described in Section 3.8.2.
A potential conflict was identified between FR2.2 (anonymization) and the requirement for group-level trend analysis: if groups are too small, even aggregated statistics could theoretically identify individual members through inference. This conflict is resolved by enforcing a minimum group size of five members before any group-level feature computation is performed, and by suppressing forecast generation for groups below this threshold until sufficient membership is recorded.

3.8.2 Control Flow and Process Design
The system's operations follow four sequential processes that move data from member submission through to pastoral action, as illustrated in the DFD and sequence diagrams below.
Data Ingestion and Storage. 
A church administrator exports historical attendance records from the church's existing digital tool (e.g., Google Sheets, a spreadsheet, or a CMS) in CSV or Excel format, and uploads the file through the React interface. The Node.js API validates the upload structure, checks for required columns, enforces date format consistency, strips any columns containing personal identifiers, and aggregates records at the group level before writing cleaned records to PostgreSQL. A validation report is returned to the administrator confirming the number of records accepted and any rows flagged for review
.
Feature Engineering and Model Training. 
After each new upload, the feature engineering pipeline is triggered. The pipeline retrieves the group's full chronological record set from PostgreSQL, computes four lagged features — rolling 4-week attendance average, 8-week linear trend slope, weekly participation rate, and feedback average where available — and writes the computed features to the features table. On the first Sunday of each month, the full training pipeline executes: the accumulated features table is read, an 80/20 temporal train-test split is applied, GridSearchCV with TimeSeriesSplit cross-validation tunes the SVR hyperparameters, and the trained model and its associated StandardScaler are serialized to disk using joblib.
Forecast Generation and Alert Creation. When a pastor accesses the Group Detail page for a specific cohort, the React frontend calls the Node.js forecast endpoint. The API retrieves the group's most recently computed features, forwards them to the Python FastAPI microservice's POST /predict endpoint, and receives a JSON response containing the predicted headcount, confidence interval lower bound, and confidence interval upper bound for the requested horizon. The API persists this forecast to the forecasts table and evaluates whether the predicted headcount represents a decline of more than 15% relative to the current rolling average; if so, it creates a corresponding alert record in the alerts table with a contextual plain-language message and pastoral recommendation.
Dashboard Visualization and Action. The pastoral dashboard renders a Recharts ComposedChart displaying 12 weeks of historical attendance as a solid line and the forecast as a dashed line with a shaded confidence cone. If an active, unacknowledged alert exists for the group, an amber banner is displayed at the top of the Group Detail page containing the alert message and the recommended pastoral intervention. The pastor can dismiss the alert once action has been taken, which sets the acknowledged flag on the alert record and removes the banner.

3.8.3 SVR Model Specification and Validation Strategy
This section specifies the Support Vector Regression model configuration, kernel selection rationale, hyperparameter tuning approach, training data strategy, and validation methodology. This level of methodological specification is necessary to distinguish the proposed system from a descriptive analytics tool and to satisfy the empirical evaluation requirements of this project.
Kernel Selection Rationale
The Radial Basis Function (RBF) kernel was selected based on three evidence-based criteria. First, the RBF kernel is the standard choice for datasets with non-linear, non-monotonic relationships between features and outcomes — a characteristic of church attendance data, where satisfaction ratings and attendance interact non-linearly depending on seasonal context. Second, prior SVR applications to analogous sparse social datasets (Al-Rahmi et al., 2024; Shatte et al., 2021) uniformly reported superior accuracy with RBF over linear or polynomial kernels when training set sizes were under 200 observations. Third, the RBF kernel requires only two hyperparameters (C and gamma), reducing overfitting risk during tuning compared to polynomial kernels
.
Hyperparameter Specification and Tuning
Three SVR hyperparameters require tuning prior to model deployment:
C (regularization parameter): Controls the trade-off between maximizing the margin and minimizing training error. A higher C value reduces bias but increases variance risk on small datasets. Search range: C 
∈
 {1, 10, 100}.
γ (gamma — RBF kernel bandwidth): Controls the influence radius of each training example. A smaller γ creates a smoother decision boundary. Search values: γ 
∈
 {'scale', 'auto', 0.1, 0.01}.
ε (epsilon — insensitive loss tube): Defines the threshold below which prediction errors are ignored. This parameter directly models the subjective noise in self-reported feedback ratings. Search range: ε 
∈
 {0.05, 0.1, 0.5}.
Hyperparameters will be optimized using an exhaustive grid search (sklearn.model_selection.GridSearchCV) with Mean Absolute Error as the scoring metric, as MAE is more interpretable in the pastoral context than RMSE (an error of ±5 members is a 
meaningful pastoral threshold regardless of its squared magnitude). The grid search will evaluate all 3 × 4 × 3 = 36 parameter combinations per group model.
Training Data Strategy
The 
The SVR model will be trained on historical records sourced from five to ten partner churches in Nairobi County, supplemented by simulated longitudinal data for the initial prototype phase where real records are not yet available. Simulated data will be generated for five congregation groups over a 24-month period (104 weekly records per group, totaling 520 training observations). Simulation parameters are grounded in empirical benchmarks: baseline attendance drawn from a normal distribution calibrated against KNCC (2024) median congregation size data for Nairobi; seasonal trend components added for Easter, Christmas, and school-term boundaries; and engagement drops modeled as step-function reductions of 15 to 25% over 4 to 8 weeks, consistent with Chaves and Anderson (2022) documented disengagement trajectories.
 
These parameters ensure the simulated dataset reflects real-world distributional properties rather than arbitrary random noise.
Train-Test Split and Cross-Validation
Model training will employ a temporal (walk-forward) split to respect the time-ordered nature of the data and prevent data leakage — a critical requirement for time-series models where shuffled splits produce artificially inflated accuracy estimates. Specifically:
Training set: the first 80% of each group's chronologically ordered records (approximately 83 weeks of the 104-week simulated dataset per group).
Test set: the final 20% of records (approximately 21 weeks), used exclusively for holdout evaluation and never exposed to the model during training or hyperparameter tuning.
Cross-validation: TimeSeriesSplit with k = 5 folds applied within the training set during grid search, using expanding windows to simulate realistic deployment conditions.
This walk-forward validation strategy is consistent with the methodology employed in comparable SVR time-series studies (Zhang et al., 2023; Khan et al., 2022) and is required to produce evaluation metrics that are comparable to published benchmarks.
Baseline Comparison Models
To establish whether SVR offers meaningful predictive advantage over simpler approaches, the model is evaluated against two baseline comparators on the same holdout test set: OLS Linear Regression (using the same four engineered features as the SVR model), and ARIMA(1,1,1) (using only the historical attendance sequence). The primary evaluation metric is MAE, with a target of SVR MAE at or below 80% of OLS MAE (consistent with Al-Rahmi et al., 2024). The secondary metric is RMSE. Both metrics will be reported in Chapter 4 alongside a paired t-test on per-group error distributions to confirm statistical significance.

3.8.4 Design for Non-Functional Requirements
Security requirements are addressed through three complementary mechanisms. Authentication is implemented using JSON Web Tokens with a seven-day expiry, verified by middleware applied to every protected API endpoint. Role-based access control ensures that analyst credentials access model metrics routes only, pastor credentials access dashboard and forecast routes, and admin credentials access all routes plus user management. Parameterized queries via Prisma ORM prevent SQL injection on all database operations, and all data transmissions are secured over HTTPS with TLS 1.2 or higher on the Render deployment platform.
Reliability is addressed through graceful degradation mechanisms. If the Python SVR microservice fails to return a prediction for example because the group has insufficient historical data to have generated a trained model the Node.js API defaults to a simple linear regression forecast rather than returning an error to the dashboard, ensuring that pastors always receive a usable output. All forecast generation events and pipeline execution runs are logged to a timestamped audit log table that is append-only at the API layer, preventing tampering with historical records.
Performance 
optimization for low-bandwidth environments is addressed at both the frontend and backend layers. The React frontend uses lazy loading for chart components, reducing initial load time. The PostgreSQL schema includes composite indexes on (group_id, week_date) fields of the 
attendance and features tables, accelerating range queries required for feature retrieval. Computationally intensive operations feature computation and monthly model retraining execute during off-peak hours to avoid competing with user-facing requests
.
3.9 Physical Design
3.9.1 Database Design
The physical database schema is implemented using PostgreSQL 15 via Prisma ORM for type-safe migrations and query generation. The schema is fully normalized to third normal form and structured around congregation groups as the central organizing entity, reflecting the system's group-level analysis philosophy. Seven core tables are defined: users stores hashed credentials and role assignments; groups defines congregation cohorts with category labels and minimum size thresholds; attendance stores weekly headcounts indexed by (group_id, week_date) with a unique constraint preventing duplicate records; feedback stores aggregated weekly satisfaction scores where available; features stores the four computed lagged variables per group per week; forecasts stores SVR-generated predictions with confidence interval bounds; and alerts stores derived alert records with message, recommendation, and acknowledgment status.
The architectural enforcement of the anonymization requirement is visible in the schema: the features, forecasts, and alerts tables contain only group_id foreign keys — no member_id exists in any analytical table. The users table is completely separate from the analytical pipeline, used exclusively for authentication and role management. This schema-level separation ensures the non-profiling guarantee is architecturally enforced rather than merely policy-dependent

3.9.2 User Interface Design
The user interface is designed around three principles derived from requirements elicitation: no training required for church staff; forecasts presented in plain language rather than statistical terms; and mobile-first layout for users accessing the system via smartphones. The interface is implemented with React.js 18 and Tailwind CSS, producing a responsive layout adapting across smartphone, tablet, and desktop viewports. The color scheme uses amber for alert banners, teal for positive trend indicators, and neutral gray for historical data, providing immediate visual distinction between alert states without relying on color alone — satisfying the WCAG 2.1 AA accessibility requirement.
The four primary screens are: the Login Screen, accepting email and password credentials and routing the user to their role-appropriate starting view; the Data Upload Screen, a clean drag-and-drop interface where administrators upload CSV or Excel exports with a validation preview before submission; the Home Dashboard, displaying a summary card for each congregation group showing last headcount, 4-week trend direction, and unread alert count; and the Group Detail page, displaying the 12-week historical and forecast Recharts line chart, the active alert banner if present, and the recommendation card with a Dismiss action
