1. Executive Overview
The Faith Organization Growth Forecasting System (CG Forecast) is a full-stack web application that enables church leadership to monitor attendance, collect qualitative feedback, and receive AI-generated growth predictions for all registered faith groups. The machine learning layer uses Support Vector Regression (SVR) trained on historical attendance data to predict future attendance trajectories and automatically raise alerts when declining trends are detected.

This document serves three purposes simultaneously:
It is the complete functional specification for the Antigravity engineering team to implement, test, and verify all features.
It is the presentation script for the panel demo — describing the expected system behaviour at every step.
It is the QA checklist — each functional requirement includes a testable acceptance criterion.

1.1 System Purpose
Church leadership (pastors, administrators) currently track attendance manually — spreadsheets, paper registers, or disconnected apps. There is no mechanism to detect early warning signs of member disengagement, and no data-driven basis for pastoral intervention decisions. CG Forecast solves this by:
Centralising data collection: Ushers log headcounts and RSVPs digitally at the point of service.
Quantifying engagement: Feedback ratings are collected alongside attendance to produce a composite engagement score.
Automating intelligence: An SVR model trained per faith group predicts attendance for the next 6 weeks and alerts leadership when a decline pattern is detected with high confidence.
Enabling role-based access: Pastors and admins see forecasts and alerts; ushers log data; members view their own group. Each role sees only what is relevant to them.

6. Machine Learning Engine — SVR Forecasting
This is the core differentiator of the system. Each faith group has its own dedicated SVR (Support Vector Regression) model, trained on that group's historical attendance and feedback data. The model predicts attendance for the next 6 sessions (approximately 6 weeks for weekly groups).

6.1 How the ML Pipeline Works
Training Trigger Events
The SVR model is automatically retrained in four situations:
A new attendance record is submitted (triggers async retraining in background)
New feedback is submitted for a group (retraining with updated feature weights)
A manual retrain is triggered by admin from the Settings > ML Models page
A scheduled nightly job runs at 02:00 local time to retrain all models on latest data

Training Data Requirements

< 5 attendance records for group


5–14 records
Model trained but confidence interval is wide (±20%)



30+ records

Act 3 — Forecast Deep Dive (3 min)

12. Appendix — Data Models
These are the canonical data structures that the frontend expects from the API. Any deviation from these shapes will cause UI rendering errors.

12.1 Core API Response Shapes
// User
{ id: string, name: string, email: string,
  role: "admin" | "pastor" | "usher" | "member" }
 
// Group
{ id: number, name: string, category: string,
  leader: { name: string }, createdAt: string,
  memberCount: number, status: "active"|"warning"|"alert" }
 
// Attendance Record
{ id: number, groupId: number, sessionDate: string,
  headcount: number, rsvpCount: number | null,
  notes: string | null, loggedBy: string }
 
// Forecast Data Point (array of 10 = 4 historical + 6 predicted)
{ date: string, actual: number | null,
  predicted: number | null,
  lowerBound: number | null, upperBound: number | null }
 
// Alert
{ id: number, groupId: number,
  alertType: "drop" | "growth" | "stable",
  severity: "critical" | "warning" | "positive" | "info",
  message: string, recommendation: string,
  acknowledged: boolean, createdAt: string }
 
// Feedback
{ id: number, groupId: number, sessionDate: string,
  avgRating: number, respondentCount: number,
  themes: string | null, loggedBy: string }
